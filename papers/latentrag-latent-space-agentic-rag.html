LatentRAG: Latent-Space Reasoning for Efficient Agentic RAG

NLP / Efficiency Research Group · 2026-05

RAG · 潜空间推理 智能体 · 多跳问答 arXiv 2605.06285 ↗

TL;DR

LatentRAG 将检索与推理统一到共享的连续潜空间中，通过双编码器 + 交叉注意力融合将检索段落表示直接注入模型推理状态，彻底消除了迭代 RAG 中各轮检索之间的 token 生成往返开销。多跳 QA 任务延迟降低 90%，LLM 调用次数减少 4.2x，同时保持与迭代 RAG 相当的准确率。

研究背景

  • 迭代 RAG 延迟瓶颈：标准迭代 RAG 在每轮检索后需要完整生成 token 序列作为下一轮检索的查询，每次 LLM 调用都引入完整的前向传播延迟
  • 多跳推理的级联开销：多跳 QA 需要多轮检索，迭代方式导致延迟与跳数线性增长
  • 检索-生成语义不匹配：token 级接口限制了检索段落与模型内部推理状态的深度融合
  • 智能体 RAG 效率：在长链条智能体工作流中，RAG 的调用延迟成为整体系统的主要瓶颈

核心方法

1. 双编码器潜空间统一：

// 查询编码器：将当前推理状态映射到检索空间 h_q = QueryEncoder(reasoning_state) // 段落编码器：将检索段落映射到同一潜空间 h_p = PassageEncoder(retrieved_passage) // 相似度检索 scores = sim(h_q, h_p) // 余弦或点积

2. 交叉注意力融合：将检索段落的表示向量直接通过交叉注意力层注入模型的推理状态，无需将段落解码为 token 序列再重新编码，实现零往返延迟的检索融合。

// 交叉注意力融合（无 token 往返） new_state = CrossAttention( query=reasoning_state, key=h_p, value=h_p ) // 直接更新推理状态，进入下一跳检索

3. 端到端联合训练：检索器和推理模型联合训练，对齐潜空间表示，确保融合后推理状态质量。

实验结果

  • 多跳 QA 延迟：vs 迭代 RAG 基线降低 90%
  • LLM 调用次数：减少 4.2x（多跳推理中无需每跳调用完整 LLM 生成中间 token）
  • 准确率：在 HotpotQA、MuSiQue 等多跳 QA benchmark 上与迭代 RAG 持平
  • 智能体工作流：在长链条智能体任务中整体系统延迟显著降低

核心贡献

  1. 首次将检索与推理统一在连续潜空间，消除 token 往返延迟
  2. 双编码器 + 交叉注意力融合实现零开销的多跳检索状态更新
  3. 在保持准确率的前提下将多跳 QA 延迟降低 90%
  4. 为智能体 RAG 工作流提供了高效的系统设计范式

局限性

  • 需要端到端联合训练，与现有预训练 LLM 的即插即用集成较复杂
  • 潜空间融合可能损失部分原始段落的细粒度文本信息
  • 双编码器的训练数据需要多跳推理路径监督信号，标注成本较高
  • 在开放域单跳检索场景下的收益相对有限
