已有方法的局限性

  • 效率度量维度错误：传统研究聚焦于参数量压缩（量化、剪枝、蒸馏），忽视了 output token 数对推理延迟的决定性影响。
  • CoT 带来的效率悖论：Chain-of-Thought 显著提升了准确率，但同时大幅增加了 output token 数，大模型的实际延迟远超预期。
  • 现有协作方案方向错误：Speculative Decoding 关注「加速 token 生成速度」，而非「减少大模型需要生成的 token 总量」——治标不治本。

本文核心问题

能否让大模型不生成完整推理链，而是「借阅」小模型的推理过程，只输出简短最终答案，同时保持大模型的准确率优势？

⚙️ 核心方法

整体框架

首先通过系统性延迟分析实验，建立 output token 数与延迟的定量关系，证明其线性相关性远强于参数量与延迟的关系——这是全文的核心观察。在此基础上，提出多智能体推理框架：小模型（如 7B）负责生成完整的 CoT 推理链，大模型（如 70B）接收小模型推理 token 作为上下文，仅生成最终短答案。

关键模块

① 延迟归因分析框架：系统测量不同规模模型在不同推理长度下的端到端延迟，建立定量关系，证明 output token 数是主瓶颈，而非参数量。这是反直觉的核心发现。
② 推理 Token 复用机制：小模型输出完整 CoT → 大模型将小模型推理 token 拼接至输入上下文 → 大模型在「阅读」小模型推理后只生成短答案。大模型的知识容量用于校验和补全，而非重新推导。
③ 动态任务分配：根据任务复杂度决定是否启用推理 token 复用，以及选择合适规模的小模型作为推理代理，平衡精度-延迟 trade-off。

关键公式

设大模型单 token 延迟为 α，小模型为 α'，则：
传统大模型延迟：T_large ≈ α · N_out_large
协作模式延迟：T_collab = T_small + α · N_ans
其中 N_ans ≪ N_out_large，实现延迟大幅降低

与已有方法的本质区别

Speculative Decoding 是「用小模型加速大模型的 token 生成」；本文是「用小模型替代大模型的推理过程，让大模型只做最终判断」。方向完全不同——前者优化速度，后者减少总量。与蒸馏相比，本文不修改任何模型权重，完全是推理时的部署策略。

  • 错误传播风险：小模型推理链若有误，大模型可能被错误逻辑误导，错误传播机制未被充分讨论。
